## Architecture Overview

All initiatives are independent Spring Boot + React applications that share **one external PostgreSQL database** (AWS RDS/localhost/container). They discover each other and all shared services through a signed **user** and their various accesses in the database by the Admin in the Admin Console.

```
┌─────────────────────────────────────────────────────────┐
│                   Admin Console           │
│  • JWT issuer for all user auth                         │
│  • Connects to Database → tables auto-created            │
│  • Generates Connect Token → distributed to initiatives │
│  • Manages tenants, LLM config, risk profiles           │
└────────────────────┬────────────────────────────────────┘
                     │  Connect Token (signed JWT)
          ┌──────────┼──────────────────┐
          ▼          ▼                  ▼                   
   DD-Engine     Cloud-infra    Other Initiatives yet to be made
  
```

### Connect Token

The Connect Token is a **signed JWT** (HS256, same `JWT_SECRET` across all services) generated by the Admin Console. It contains everything an initiative needs to operate:

```json
{
  "db_jdbc_url":        "jdbc:postgresql://<rds-host>:5432/pas_db",
  "db_user":            "pas",
  "db_password":        "<password>",
  "minio_endpoint":     "http://localhost:$chosen_host",
  "minio_bucket":       "pas-documents",
  "minio_access_key":   "pas_minio",
  "minio_secret":       "pas_minio_secret",
  "meilisearch_url":    "http://localhost:$chosen_host",
  "meilisearch_key":    "pas_meili_local",
  "libretranslate_url": "http://localhost:$chosen_host",
  "paddleocr_url":      "http://localhost:$chosen_host",
  "llm_provider":       "groq",
  "llm_model":          "llama-3.3-70b-versatile",
  "llm_api_key":        "<groq-key>",
  "initiatives":        ["admin-console", "alphacore", "autoops","cloud-infra","cybersecurity", "data-warehouse", "dd-engine", "deal-flow", "ic-workflow", "investor-crm", "investor-portal", "investor-reporting", "knowledge-base", "market-intelligence", "portfolio-monitor", "risk-analytics"]
  "exp":                "<unix timestamp>"
}
```

Each initiative exposes `POST /api/connect` — send the token there and it self-configures.

## Environment Variables

Every initiative reads the same set of env vars. The only one **required at startup** is `JWT_SECRET` — everything else can be configured later via the Admin Console UI.

| Variable           | Required | Default                          | Description                        |
|--------------------|----------|----------------------------------|------------------------------------|
| `JWT_SECRET`       | ✅ Yes   | —                                | Shared signing key — same on ALL services (min 32 chars) |
| `DB_URL`           | No       | *(set via Admin Console UI)*     | JDBC URL to AWS RDS                |
| `DB_USER`          | No       | *(set via Admin Console UI)*     | RDS username                       |
| `DB_PASS`          | No       | *(set via Admin Console UI)*     | RDS password                       |
| `MINIO_ENDPOINT`   | No       | `http://localhost:9000`          | MinIO / R2-compatible storage      |
| `MINIO_BUCKET`     | No       | *(set via Admin-console/Cloud)*  | Object storage bucket              |
| `MINIO_ACCESS_KEY` | No       | *(to be convened)*               | MinIO access key                   |
| `MINIO_SECRET`     | No       | *(to be convened)*               | MinIO secret                       |
| `MEILI_URL`        | No       | *(to be convened)*               | Meilisearch URL                    |
| `MEILI_KEY`        | No       | *(to be convened)*               | Meilisearch master key             |
| `LIBRETRANSLATE_URL` | No     | *(to be convened)*               | LibreTranslate URL                 |


Open **http://localhost:5200**

Default login:
- Email: `admin@primealphasecurities.com`
- Password: `admin123`

**First-time setup in the UI:**
1. Go to **Database** tab → enter your AWS RDS credentials → **Connect to RDS** (tables auto-created)
2. Go to **LLM Config** tab → select provider + paste API key → **Save**
3. Go to **Connect** tab → **Generate token** → copy the JWT

### Step 2 — Distribute the Connect Token

Each initiative has a `POST /api/connect` endpoint. Send the token once:

```bash
TOKEN="<paste-token-from-admin-console>"

# ✅ DD-Engine
curl -X POST http://localhost:$chosen_port/api/connect \
  -H "Content-Type: application/json" \
  -d "{\"token\": \"$TOKEN\"}"

```

After that, each initiative is fully configured. No restart needed.

### Step 3 — Run Initiatives

#### For all Initiatives

```bash
# Terminal 3 — Backend 
source /home/johan/PAS/Tech/PROJECT/.env
cd /home/johan/PAS/Tech/PROJECT/$initiative/backend
mvn spring-boot:run

# Terminal 4 — Frontend 
cd /home/johan/PAS/Tech/PROJECT/$initiative/frontend
npm run dev
```

## Supporting Services (optional, for full features)

These are only needed for file storage (MinIO), search (Meilisearch), translation (LibreTranslate), and OCR (PaddleOCR). The Admin Console and AlphaCore work without them. DD-Engine needs MinIO and PaddleOCR for document processing.

**Install and run natively (no Docker):**

```bash
# MinIO — object storage (R2-compatible)
wget https://dl.min.io/server/minio/release/linux-amd64/minio
chmod +x minio
MINIO_ROOT_USER=pas_minio MINIO_ROOT_PASSWORD=pas_minio_secret ./minio server ~/minio-data --console-address :9001
# API: http://localhost:9000  Console: http://localhost:9001

# Meilisearch — full-text search
curl -L https://install.meilisearch.com | sh
MEILI_MASTER_KEY=pas_meili_local ./meilisearch
# http://localhost:7700

# LibreTranslate — translation
pip install libretranslate
libretranslate --load-only en,fr,ar
# http://localhost:5000

# PaddleOCR sidecar — already in shared/docker/paddleocr/server.py
pip install paddlepaddle paddleocr flask pillow
python /home/johan/PAS/Tech/PROJECT/shared/docker/paddleocr/server.py
# http://localhost:5299
```

## How Each Initiative Is Built (pattern for all)

Every initiative follows the exact same structure:

```
<initiative>/
├── backend/
│   ├── pom.xml                          # Spring Boot 3.3, Java 21, Maven
│   └── src/main/
│       ├── java/com/pas/<initiative>/
│       │   ├── <Name>Application.java
│       │   ├── config/SecurityConfig.java   # JWT filter + CORS
│       │   ├── security/
│       │   │   ├── JwtService.java          # validates tokens from Admin Console
│       │   │   └── JwtAuthFilter.java
│       │   ├── model/                       # JPA entities → tables from schema
│       │   ├── repository/                  # Spring Data JPA repos
│       │   ├── service/
│       │   │   ├── ConnectService.java      # handles POST /api/connect
│       │   │   └── ...
│       │   └── controller/
│       │       ├── ConnectController.java   # POST /api/connect
│       │       └── ...
│       └── resources/application.yml
└── frontend/
    ├── package.json                     # React 18 + Vite + axios
    ├── vite.config.js                   # proxies /api → backend port
    ├── index.html
    └── src/
        ├── main.jsx
        └── App.jsx                      # all UI in one file, DD-Engine design tokens
```

**JWT flow:**
1. User logs in via Admin Console → gets a JWT signed with `JWT_SECRET`
2. Frontend sends `Authorisation: Bearer <token>` on every request
3. Each initiative's `JwtAuthFilter` validates the signature using the same `JWT_SECRET`
4. No initiative issues its own tokens — Admin Console is the sole auth issuer

**Connect flow:**
1. Admin Console generates a Connect Token (also a JWT) with all service config embedded
2. Operator runs `curl -X POST http://localhost:<port>/api/connect -d '{"token":"..."}'`
3. Initiative validates the token, extracts DB URL + service URLs + LLM config, stores in memory
4. From that point the initiative is fully operational — no restart needed

## API Reference

### Admin Console 

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| POST | `/api/auth/login` | — | Login → returns JWT |
| GET | `/api/auth/me` | JWT | Current user |
| GET | `/api/db-setup/status` | — | DB connection status |
| POST | `/api/db-setup/test` | — | Test RDS credentials |
| POST | `/api/db-setup/connect` | — | Connect to RDS |
| POST | `/api/connect/generate` | JWT | Generate connect token |
| GET | `/api/connect/tokens` | JWT | List issued tokens |
| DELETE | `/api/connect/tokens/:id` | JWT | Revoke token |
| GET | `/api/tenants` | JWT | List tenants |
| POST | `/api/tenants` | JWT | Create tenant |
| GET | `/api/llm/config` | JWT | Get LLM config |
| PUT | `/api/llm/config` | JWT | Update LLM config |
